PiA - Netaxept Android SDK is a library that provides the native In-App interaction of performing the Netaxept payment directly from an app on the Android device and minimizes PCI DSS requirements for you.

Minimum compatible Android version is 5.0.
Read below for more information on how to use it.

Terminology

  1. Nets PiA SDK - The SDK. From your reference, a compiled Android library AAR.
  2. Netaxept - Netaxept is a comprehensive online payment service provided by Nets. It enables merchants to offer their customers different ways to pay purchases easily and securely in webshops, mobile environment and call centre services. When using Netaxept, it is easy to pay with broad range of payment methods and currencies, and still meet high security requirements set for online paying.
  3. Easy Payment - After the initial transaction, the customer can initiate the following transaction by logging in to your webshop or mobile app and paying simply with one click and entering their cards CVV/CVC security code. Easy Payment is applicable especially for mobile applications where it significantly eases and accelerates the returning customers purchase experience.
  4. Recurring Payments - After the initial transaction, you as a merchant can charge the customers preferred card at specific times based on the agreement you have made with th customer. Traditionally the concept of Recurring Payments is applicable especially for paying subscriptions, membership fees and other regular payments, without the need of customer interaction after the initial payment. However, in PiA SDK we use the Recurring Payments concept to allow you as a merchant to enable easy subsequent payment for returning customers, without the need to collect again their security code (CVV/CVC).

Getting started with card payment

It is as simple as creating an instance of PiA SDK and calling PiaSDK.start() with the necessary parameters based on your payment flow to allow the SDK to start populating payment activity PiaActivity.

Note The SDK can be used in three different functionalities, based on the content sent in the Bundle of PiaSDK.start(), as follows:

  • Register Card - Bundle containing:
    • MerchantInfo under key PiaActivity.BUNDLE_MERCHANT_INFO
  • Basic Payment - Bundle containing:
    • MerchantInfo under key PiaActivity.BUNDLE_MERCHANT_INFO
    • OrderInfo under key PiaActivity.BUNDLE_ORDER_INFO
  • Easy Payment - Bundle containing:
    • MerchantInfo under key PiaActivity.BUNDLE_MERCHANT_INFO
    • OrderInfo under key PiaActivity.BUNDLE_ORDER_INFO
    • TokenCardInfo under key PiaActivity.BUNDLE_TOKEN_CARD_INFO

Note: For Xamarin integration, due to obfuscation in closed-source library, the Xamarin Bindings cannot convert the Keys from PiaActivity to be public. In order to have this library fully compatible with Xamarin, we added these properties in PiaSDK class:

  • PiaSDK.BundleMerchantInfo
  • PiaSDK.BundleOrderInfo
  • PiaSDK.BundleTokenCardInfo
  • PiaSDK.BundleCompleteResult

After this step, the SDK will interact with the user to post payment sensitive data to Netaxept to process the transaction. As a consequence, you will be notified by the SDK about the PiaResult as Intent object Intent in the onActivityResult callback from your activity where the SDK is started.

If you receive the successful result PiaResult, you could follow the sequence diagram to proceed the corresponding Netaxept call(s) to finalize the transaction. Otherwise, you need to handle the cancellation or failure case accordingly.

Basic payment flow description

Below you can find the interaction between the application that uses the PiA SDK and the SDK to start the basic payment flow by using a payment card:

In the basic payment flow, PiA SDK should be initialized by calling PiaSDK.start() with the reference of your activity where PiA SDK is started, bundle and RegisterPaymentHandler.

In the bundle, MerchantInfo and OrderInfo are required to allow PiA SDK to have necessary information to initialize the activity.

With regards to the RegisterPaymentHandler, this is implemented by the App to allow PiA SDK to initialize the Netaxept Register call on behalf of your app to ease of integration. You can check sample code of RegisterPaymentHandler below to determine how to construct the register call handler forPiA SDK to utilize:

public interface RegisterPaymentHandler {
    /**
     * This function will run on different thread than UI thread.
     *
     * @param saveCard Specify if the card should be saved.
     * @return The transaction id received as a result of the register payment request.
     */
    TransactionInfo doRegisterPaymentRequest(boolean saveCard);
}
public class RegisterPaymentHandlerImpl implements RegisterPaymentHandler {
    @Override
    public TransactionInfo doRegisterPaymentRequest(boolean saveCard) {
        PaymentFlowCache paymentFlowCache = PaymentFlowCache.getInstance();
        PaymentRegisterRequest paymentRegisterRequest = paymentFlowCache.getPaymentRegisterRequest();
        paymentRegisterRequest.setStoreCard(saveCard);

        MerchantRestClient.getInstance().registerPayment(paymentRegisterRequest);

        if (!paymentFlowCache.finishedWithError()) {
            PaymentRegisterResponse registerResponse = paymentFlowCache.getPaymentRegisterResponse();
            return new TransactionInfo(registerResponse.getTransactionId(),
                    registerResponse.getRedirectOK(),
                    registerResponse.getRedirectCancel());
        }

        return null;
    }
}

The code sample below renders the example of how an initialization procedure of PiA SDK could be performed:

public class MainActivity extends AppCompatActivity {

    private RegisterPaymentHandler mRegisterPaymentHandler;
    private static final String CURRENCY = "EUR";
    private static final String ORDER_NUMBER = "PiaSDK-Android";

    @Override
    protected void onCreate(Bundle savedInstanceState) {
        super.onCreate(savedInstanceState);
        mRegisterPaymentHandler = new RegisterPaymentHandlerImpl();

    }

    private MerchantInfo getMerchantInfo() {
        return new MerchantInfo(BuildConfig.MERCHANT_ID);
    }

    private OrderInfo getOrderInfo() {
        String priceString = mPriceView.getText().toString();
        double price = priceString.isEmpty() ? 0 : Double.parseDouble(priceString);

        return new OrderInfo(
        price,
        CURRENCY
        );
    }

    private void callPiaSDK(PaymentMethod method) {
        Bundle bundle = new Bundle();
        bundle.putParcelable(PiaActivity.BUNDLE_MERCHANT_INFO, getMerchantInfo());
        bundle.putParcelable(PiaActivity.BUNDLE_ORDER_INFO, getOrderInfo());
        PiaSDK.getInstance().start(MainActivity.this, bundle, mRegisterPaymentHandler);
    }
}

After the transaction has been registered, an unique transaction identifier (transaction id) in TransactionInfo will be returned from Netaxept to PiA SDK and it will post it along with the card data to the Netaxept. In addition, two redirect URLs are required by PiA SDK so that the SDK can be notified by Netaxept after payment card data have been delivered to Netaxept successfully. During this phase, the 3DS authentication is performed as well unless it has been waived by you or your chosen card acquirer.

In the end, from your activity where PiA SDK is initialized, you need to override the onActivityResult callback to be notified when the SDK handling is completed. You can find the SDK result stored as Intent object.

The code sample below indicates the example of how to override the callback to receive the SDK result:

public class MainActivity extends AppCompatActivity {

    {@literal @}Override
    protected void onActivityResult(int requestCode, int resultCode, Intent data) {

        if (requestCode != PiaSDK.PIA_SDK_REQUEST) {
            super.onActivityResult(requestCode, resultCode, data);
        }

        //in case user cancelLed
        Bundle bundle = new Bundle();
        if (resultCode == RESULT_CANCELED) {
            rollbackTransaction();
            return;
        }

        if (resultCode == RESULT_OK) {
            PiaResult result = data.getParcelableExtra(PiaActivity.BUNDLE_COMPLETE_RESULT);
            if (result.isSuccess()) {
                //in case of success, commit the payment
                mRestClient.commitPayment(mPaymentCache.getPaymentRegisterResponse().getTransactionId());
            } else {
                // Otherwise, handle the failure case
            }
        }
    }
}

3DS Basics
If the chosen card acquirer has the 3DS verification required, this process will be triggered in a WebView.
In the case 3DS is not required, the transaction is performed directly.
PiA SDK also includes support for Nordea Visa Cards, which have an extra option inside the 3DS process: BankID Verification.
If the user chooses the BankID option from the WebView, the BankID application will be opened automatically to confirm the payment.
After the BankID authentication is complete, when returning back to the SDK, the web page will detect if the verification was performed and returns a proper result. If the user does not have
the BankID app installed on his device, he will be prompted to Play Store to install it. After that, he can resume the 3DS process.

Easy payment flow description

After the initial transaction, the customer can initiate the following transaction by logging into your webshop or mobile app and paying simply with one click and entering their cards CVV/CVC security code. Easy Payment is applicable especially for mobile applications where it significantly eases and accelerates the returning customers purchase experience.

Below you can find the interaction between the application that uses the PiA SDK and the SDK for this transaction flow:

Initial transaction of Easy Payment flow
In the initial transaction of Easy Payment flow, PiA SDK should be started with the the basic payment flow. Then, Easy payment can be triggered if the user selects the option to save the card for future usage during the card entry process. In this case, the SDK will pass the boolean parameter saveCard with true(by default, it is set to false to trigger the basic payment flow as mentioned above) in the RegisterPaymentHandler. This instructs the app to go ahead with the Easy payment flow and you need to make the transaction registration according to the sequence diagram above.

After your app gets the successful result from PiA SDK, you can start Netaxept call(s) to proceed the transaction and fetch and store token in your backend which will be utilized for the subsequent transaction of Easy Payment.

Subsequent transaction of Easy Payment flow
In the subsequent transaction of Easy payment, you can follow the same way as the initial transaction to start PiA SDK with just one extra dataTokenCardInfo in the bundle to provide the token information for PiA SDK to present token card and convey it to Netaxept for charging the transaction. Depending on your settings and your agreement with your chosen acquirer you may or may not have to ask the customer for the CVV/CVC security code for each transaction. Easy Payment on Netaxept side supports both variants. Certain industries have an exception from using the security code so please consult your chosen acquirer for further information if needed. You can set the boolean parameter cvcRequired from TokenCardInfo to notify PiA SDK to ask for the security code from the customer. If the security code is not required, another optional security method can be used: the system authentication feature. With this feature, the user will use the unlock screen functionality to confirm his identity (pin, password, pattern or fingerprint) for subsequent payments.

The snippet below illustrates how to start the PiA SDK for subsequent transaction of Easy Payment with the extra data TokenCardInfo, and security code (CVV/CVC) required: (systemAuthRequired flag is configurable to enable/disable the system authentication for the subsequent transaction. For the Easy payment, the systemAuthRequired can be true to enable the system authentication if CVV/CVC is NOT required. However, if the CVV/CVC is required, the system authentication will be skipped anyway to avoid extra customer interaction)

//region global params
private boolean cvcRequired = true;
private boolean systemAuthRequired = false;
//end region
            
private void callPiaSDK(PaymentMethod method){
    Bundle bundle=new Bundle();
    bundle.putParcelable(PiaActivity.BUNDLE_MERCHANT_INFO,getMerchantInfo());
    bundle.putParcelable(PiaActivity.BUNDLE_ORDER_INFO,getOrderInfo());
    bundle.putParcelable(PiaActivity.BUNDLE_TOKEN_CARD_INFO,getTokenizedCardInfo((DisplayedToken)method));

    PiaSDK.getInstance().start(this,bundle,mRegisterPaymentHandler);
    overridePendingTransition(R.anim.slide_in_bottom,R.anim.slide_out_bottom);

    mPaymentCache.setState(PaymentFlowState.CALL_PIA_SDK);
}

private TokenCardInfo getTokenizedCardInfo(DisplayedToken method){
    return new TokenCardInfo(method.getTokenId(), SchemeType.VISA, method.getExpiryDate(), cvcRequired, systemAuthRequired);
}

Easy (one-click) payment without Security code (CVV/CVC) flow description

In PiA SDK, we use the Recurring Payments concept to allow you as a merchant to enable easy subsequent payment for returning customers, without the need to collect again their security code (CVV / CVC).

Below you can find the interaction between the application that uses the PiA SDK and theSDK for this transaction flow:

Initial transaction of Easy (one-click) payment without security code (CVV/CVC) flow
In the initial transaction of Easy (one-click) payment without security code flow,PiA SDK should be started like the basic payment flow. Then, Easy payment can be triggered if the user chooses to save the card for the future usage during the card entry process. Then, the SDK will pass the boolean parameter >saveCard with true(by default, it is set to false to trigger the basic payment flow as mentioned above) in the RegisterPaymentHandler to notify the app to go ahead with the Easy payment flow. You shall make the transaction registration according to the sequence diagram above.

After your app gets the successful result from PiA SDK, you can start Netaxept call(s) to proceed the transaction and fetch and store token in your backend which will be utilized for the subsequent transaction of Recurring Payment.

Subsequent transaction of Easy (one-click) payment without security code (CVV/CVC) flow
In the subsequent transaction of Easy (one-click) payment without security code, you can follow the same way as the initial transaction to start PiA SDK with just one extra data TokenCardInfo in the bundle to provide the token information for PiA SDK to present token card and convey it to Netaxept for charging the transaction.

Depending on your requirements, for subsequent payments you can customize the flow:

  1. Ask for System Authentication feature as security check (pin, password, pattern or fingerprint). You can set the boolean parameter systemAuthRequired = true from TokenCardInfo. To use this functionality, the cvcRequired flag must be false.
    Sample:
//region global params
private boolean cvcRequired = false;
private boolean systemAuthRequired = true;
//end region

private void callPiaSDK(PaymentMethod method){
    Bundle bundle=new Bundle();
    bundle.putParcelable(PiaActivity.BUNDLE_MERCHANT_INFO,getMerchantInfo());
    bundle.putParcelable(PiaActivity.BUNDLE_ORDER_INFO,getOrderInfo());
    bundle.putParcelable(PiaActivity.BUNDLE_TOKEN_CARD_INFO,getTokenizedCardInfo((DisplayedToken)method));

    PiaSDK.getInstance().start(this,bundle,mRegisterPaymentHandler);
    overridePendingTransition(R.anim.slide_in_bottom,R.anim.slide_out_bottom);

    mPaymentCache.setState(PaymentFlowState.CALL_PIA_SDK);
}

private TokenCardInfo getTokenizedCardInfo(DisplayedToken method){
    return new TokenCardInfo(method.getTokenId(), SchemeType.VISA, method.getExpiryDate(), cvcRequired, systemAuthRequired);
}
  1. The System Authentication feature is a native Android OS functionality, which is started from PiA SDK only if the flag cvcRequired = fals and the systemAuthRequired = true. Note that the SDK does not store any passwords or sensitive data. Hence the Android OS handles the unlock screen option that the user has set.
    When users click the PAY button, they will be asked to authenticate themselves by one of the following methods: Pin, Password, Pattern or Fingerprint. In this case, there can be three scenarios:
    • The user succeeds in authenticating himself: the payment flow will be similar to Easy Payment flow without security code.
    • The user fails to authenticate himself: the PAY button will be disabled and he will be prompted with an alert mentioning that he cannot complete the transaction with with this saved card. He will need to proceed with another payment method.
    • The user does not have any system authentication methods (Pin, Password, Pattern or Fingerprint): an alert will be shown with the option to go to Lock Screen options to enable any of the available authentication methods.
  2. One-click easy payment. No extra security check. You can set the boolean parameters cvcRequired = false and systemAuthRequired = false from TokenCardInfo.
    Sample:
//region global params
private boolean cvcRequired = false;
private boolean systemAuthRequired = false;
//end region

private void callPiaSDK(PaymentMethod method){
    Bundle bundle=new Bundle();
    bundle.putParcelable(PiaActivity.BUNDLE_MERCHANT_INFO,getMerchantInfo());
    bundle.putParcelable(PiaActivity.BUNDLE_ORDER_INFO,getOrderInfo());
    bundle.putParcelable(PiaActivity.BUNDLE_TOKEN_CARD_INFO,getTokenizedCardInfo((DisplayedToken)method));

    PiaSDK.getInstance().start(this,bundle,mRegisterPaymentHandler);
    overridePendingTransition(R.anim.slide_in_bottom,R.anim.slide_out_bottom);

    mPaymentCache.setState(PaymentFlowState.CALL_PIA_SDK);
}

private TokenCardInfo getTokenizedCardInfo(DisplayedToken method){
    return new TokenCardInfo(method.getTokenId(), SchemeType.VISA, method.getExpiryDate(), cvcRequired, systemAuthRequired);
}

If the user has not set any secure unlock option, a dialog will be prompted asking him to set one. The payment transaction will not be processed until this requirement has been completed.
After the unlock screen has been presented, if the user refuses to confirm his unlock option, or after a certain number of attempts, returning back to the SDK will disable the pay button, with a custom pop-up message.
Note: Due to Android limitations, the Smart Lock feature is not accessible for this kind of identity verification. These unlock methods fall under the Smart Lock category and are not primitive security methods (they are not enclaved in a trusted place), hence cannot be used. So, the System Authentication restricts this use only to pin, password, pattern or fingerprint.

Register flow description

Below you can find the interaction between the application that uses the PiA SDK and the PiA SDK for the Register Flow:

Note: Camera permission is handled automatically by the Card.IO library.

The Register payment flow logic is similar to Basic payment flow, with some small differences:

  1. Only MerchantInfo object is requred in the bundle when starting the PiA SDK.
  2. When setting the PaymentRegisterRequest, make sure to specify the amount = 0 (zero).
  3. In the onActivityResult callback, if you are returned RESULT_OK, you will need to call the verifyPayment API with the received transactionId.
public class MainActivity extends AppCompatActivity {

private RegisterPaymentHandler mRegisterPaymentHandler;
private static final String CURRENCY = "EUR";
private static final String ORDER_NUMBER = "PiaSDK-Android";

@Override
protected void onCreate(Bundle savedInstanceState) {
    super.onCreate(savedInstanceState);
    mRegisterPaymentHandler = new RegisterPaymentHandlerImpl();

}

private MerchantInfo getMerchantInfo() {
    return new MerchantInfo(BuildConfig.MERCHANT_ID);
}

private void callPiaSDK(PaymentMethod method) {
    Bundle bundle = new Bundle();
    bundle.putParcelable(PiaActivity.BUNDLE_MERCHANT_INFO, getMerchantInfo());
    PiaSDK.getInstance().start(MainActivity.this, bundle, mRegisterPaymentHandler);
}

@Override
protected void onActivityResult(int requestCode, int resultCode, Intent data) {

    if (requestCode != PiaSDK.PIA_SDK_REQUEST) {
        super.onActivityResult(requestCode, resultCode, data);
    }

    //in case user cancelled
    Bundle bundle = new Bundle();
    if (resultCode == RESULT_CANCELED) {
        rollbackTransaction();
        return;
    }

    if (resultCode == RESULT_OK) {
        PiaResult result = data.getParcelableExtra(PiaActivity.BUNDLE_COMPLETE_RESULT);
        if (result.isSuccess()) {
            //in case of success, verify the payment
            mRestClient.verifyPayment(mPaymentCache.getPaymentRegisterResponse().getTransactionId());
        } else {
            // Otherwise, handle the failure case
        }
    }
}

The main flow for registering a card:

  • Scan Credit Card screen is displayed. User can scan the card, to simplify the process, or can proceed to Manual Entry screen. After scanning successfully, he will be prompted to manual entry screen, with the fields pre-populated (card number and expiration date - if were detected the Card.IO library), to check if the data is valid, and correct if necessary.
  • User fills the CVV/CVC, and click Save Card button.
  • 3D Secure WebView is prompted (3DS authentication is performed unless it has been waived by you or your chosen card acquirer). Here he can navigate inside this web page, using custom build controls (Back, Forward, Close).
  • If user closes the webview (or clicks hardware back button) he will return to the manual entry page.
  • After filling the confirmation code in 3D Secure, the user is returned back to the application, with a result code, to be handled in the OnActivityResult.

For Netaxept Production environment, when saving a card and the 3D Secure process is triggered, this is handled inside the SDK, without storing any information related to the transaction. The save card process consists in a payment towards Netaxept with 0 (zero) amount. However, when Netaxept processes this request, is forced to use the one major amount of the specific currency (e.g. for EUR currency, the major amount is 1 EUR).

Note:* This amount wont be charged to the user, this is for verifying purpose only due to requirement from the issuer.
Note:** on Android API Lever 23 or higher, the user has the possibility to revoke the permission from Settings. If the user revokes the permission while the Card Scanner is in background, then resumes it, may be an Android OS limitation of behaviour. This edge case causes the OS to kill the process of the app, forcing the activities to recreate, loosing all stored data. A workaround implemented is to cancel the transaction, and return the user back to application with RESULT_CANCELED code, and a toast specifying that something went wrong.

PayPal flow description

For a seamless experience, PayPal Express Checkout page will be loaded in a secure manner inside a WebView (very similar to the 3DS case). The user will be prompted to the login page, where he can safely enter his credentials and proceed with the authorization. As soon as the checkout finishes (success, error or cancel) he will be returned back to the application with a PiaResult object.

Initialize PayPal payment process:
It is as simple as creating an instance of PiA SDK and calling PiaSDK.startPayPalProcess() with the necessary parameters.
Note: The bundle parameter must contain only the MerchantInfo object under key PiaActivity.BUNDLE_MERCHANT_INFO.

Note: For register payment call, you can configure your BackEnd to receive optional customer related fields, and the mobile application can provide them (for PayPal example, they can serve as a predefined shipping address). For further information, please check the Netaxept Documentation.

Related to RegisterPaymentHandler implementation, receiving the PiaResult and committing or rolling back the transaction, everything is handled the same as card payment. See more in the Basic payment flow description.

The WebView which is hosting the PayPal checkout page is using cookies only during the transaction. After the transaction has ended, the cookies and cache are cleared. So, the user will need to authenticate each time he will use PayPal as payment method.

UI Customization API

PiA SDK provides a set of options to modify UI elements and match your graphical preferences. The UI customization can be done by simply assigning your customized values to the singleton object fields PiaInterfaceConfiguration.
By using this API, you can easily change/customize the background color, text color, text font and also trigger the switch to Save card for later use before calling the PiA SDK. Besides, you can also customize the Card scanning view to be different from the rest of the UI by referencing to Card IO specific options.
When applying the configuration, please keep in mind the following:

  • For colors, pass resolved colors only (e.g. Color.Red, or ContextCompat.getColor())
  • For the fonts, you can use the assets files to create TypeFace object (e.g. Typeface.createFromAsset())
  • For drawable selectors, you can create your custom XML selectors, for different button states: normal, pressed, disabled

Note: It is recommended to apply the customization in the Application class, to have the SDK configured at runtime. Any field that is not configured, will be using the default value. Check the snippet below for reference:

/**
 * As colors, pass only resolved colors (ColorInt): ContextCompat.getColor() or Color objects, as below
 */
private void customizeSDK() {
    //SDK Configuration
    PiaInterfaceConfiguration.getInstance().setLabelTextColor(ContextCompat.getColor(this, R.color.custom_orange_color));
    PiaInterfaceConfiguration.getInstance().setFieldTextColor(Color.BLUE);
    PiaInterfaceConfiguration.getInstance().setButtonTextColor(Color.WHITE);
    PiaInterfaceConfiguration.getInstance().setSwitchThumbColor(ContextCompat.getColor(this, R.color.pia_green_color));
    PiaInterfaceConfiguration.getInstance().setMainButtonBackgroundSelector(ContextCompat.getDrawable(this, R.drawable.pia_save_card_button));
    PiaInterfaceConfiguration.getInstance().setToolbarActionButtonTextColor(ContextCompat.getColor(this, R.color.colorAccent));
    PiaInterfaceConfiguration.getInstance().setLabelFont(Typeface.SANS_SERIF);
    PiaInterfaceConfiguration.getInstance().setButtonFont(Typeface.SERIF);
    PiaInterfaceConfiguration.getInstance().setFieldFont(Typeface.MONOSPACE);
    PiaInterfaceConfiguration.getInstance().setSaveCardSwitchDefault(true);
    PiaInterfaceConfiguration.getInstance().setLogoDrawable(ContextCompat.getDrawable(this, R.drawable.digital_payment_by_nets));
    PiaInterfaceConfiguration.getInstance().setErrorFieldBorderColor(Color.YELLOW);
    PiaInterfaceConfiguration.getInstance().setValidFieldBorderColor(Color.BLUE);
    PiaInterfaceConfiguration.getInstance().setToolbarBackgroundColor(ContextCompat.getColor(this, R.color.light_gray));
    PiaInterfaceConfiguration.getInstance().setToolbarTitleColor(ContextCompat.getColor(this, R.color.pia_black_color));
    PiaInterfaceConfiguration.getInstance().setBodyBackgroundColor(Color.WHITE);
    PiaInterfaceConfiguration.getInstance().setTokenCardCVCLayoutBackgroundColor(ContextCompat.getColor(this, R.color.light_gray));

    //Scan Card UI configuration
    PiaInterfaceConfiguration.getInstance().setCardIOBackgroundColor(ContextCompat.getColor(this, R.color.custom_orange_color));
    PiaInterfaceConfiguration.getInstance().setCardIOButtonTextColor(Color.RED);
    PiaInterfaceConfiguration.getInstance().setCardIOButtonTextFont(Typeface.SERIF);
    PiaInterfaceConfiguration.getInstance().setCardIOTextColor(Color.YELLOW);
    PiaInterfaceConfiguration.getInstance().setCardIOPreviewFrameColor(Color.RED);
    PiaInterfaceConfiguration.getInstance().setCardIOTextFont(Typeface.MONOSPACE);
}

Limitations

Known limitations in the SDK:

Card IO Library - Scanner

  • Non-embossed cards cannot be scanned
  • Expiry Date cannot always be scanned
  • Supported card schemes: Visa, MasterCard, American Express (15 digits)
  • Scanner preview is only supported in 640x480 resolution. In any other, the scanner fails to detect the card

Supported TLS versions

  • Due to PCI DSS requirements and known vulnerabilities in secure protocols less than TLS 1.2, TLS 1.2 is the only secure protocol available when connecting towards Netaxept. For this reason, we dont support the Android versions lower than 5.0 which are NOT enabled/supported TLS 1.2
    by default. For more information, you can check the details from Netaxept.

Disable card scanner

If you wish to disable card scanning functionality, you can toggle the following parameter within PiaInterfaceConfiguration API.

  PiaInterfaceConfiguration.getInstance().setDisableCardIO(true);

Error Codes Handling

In the following link, you can find the Microsoft Excel document with the error codes handled inside the PiA SDK, along with further description on how they are used. PiA SDK - Netaxept - Response codes

Note:*
The error codes are returned back to the application, through the onActivityResult() method, inside the PiaResult. If is error case, this object will contain thePiaError, which has getters for error code and error message. The error codes are mapped inside an enum PiaErrorCode, but each explanation message is stored in the strings file. If you want custom error messages, you can create your custom map based on the codes.
Note:** Beside the Netaxept error codes, there are handled internal SDK errors, such as:

  • GENERIC_ERROR, code 300: Terminal call has completed with an unexpected error. Please make the Netaxept Query call to get more detailed information.
  • REQUEST_FAILED, code 200: May occur in the folowing cases:
    • internet connectivity issues
    • timeout error
    • failed serialisation in models
